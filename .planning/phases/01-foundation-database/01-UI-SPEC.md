# Phase 01 — UI Design Contract

> Visual and interaction contract for Phase 1: Foundation + Database. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Scope Boundary

Phase 1 creates the Traveloop visual foundation only: route inventory, responsive app shell, global style tokens, and reusable layout primitives. It must not implement the full supplied page layouts, completed traveler workflows, or alternate screens.

Locked source order:

1. Problem statement defines features.
2. SVG wireframe defines route and flow structure.
3. Supplied visual UI images define later page layouts.
4. Brand DNA image defines shared visual language.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none |
| Preset | not applicable |
| Component library | custom React primitives |
| Icon library | lucide-react line icons |
| Font | Playfair Display for headings, Inter for UI/body, with serif and system sans fallbacks |

Manual token decision: no shadcn initialization for Phase 1. Use CSS variables in the web app global stylesheet and keep primitive components small enough to adapt to the supplied screens in later phases.

---

## Visual Foundation

Primary shell focal point: the current route content panel on an off-white page surface, framed by compact Traveloop navigation.

Visual hierarchy:

1. Traveloop wordmark or compact compass mark in the header.
2. Page title or route placeholder heading.
3. Primary action, when available, using dark olive fill.
4. Supporting cards, search bars, filters, and route inventory content.

Phase 1 route placeholders should feel intentionally scaffolded, not empty mock pages. Use restrained surface cards, subtle dividers, and route labels that match the supplied screens. Hero-scale travel photography is reserved for later page implementation phases; Phase 1 may define image container primitives but should not invent new hero layouts.

Icon-only actions must include accessible names through `aria-label` and visible text when space allows. Header icons use lucide-style 20px strokes with 44px minimum touch targets.

---

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, badge inner gaps, tight metadata separators |
| sm | 8px | Compact element spacing, chip padding, nav item gaps |
| md | 16px | Default element spacing, card padding on mobile, form control spacing |
| lg | 24px | Card padding on desktop, shell gutters on mobile, toolbar gaps |
| xl | 32px | Desktop layout gaps, section spacing inside main content |
| 2xl | 48px | Major shell content breaks, page header spacing |
| 3xl | 64px | Wide desktop page-level spacing only |

Exceptions: 44px minimum touch target for icon buttons and compact navigation controls; justified by accessibility tap target requirements.

---

## Typography

Use exactly four font sizes and two font weights for Phase 1 shell work.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 16px | 400 | 1.5 |
| Label | 14px | 500 | 1.4 |
| Heading | 28px | 500 | 1.2 |
| Display | 48px | 500 | 1.1 |

Rules:

- Body, labels, nav, controls, and metadata use Inter/system sans.
- Heading and Display use Playfair Display/serif.
- Do not scale type with viewport width.
- Letter spacing stays 0.
- Page placeholders should use Heading, not Display, unless a supplied visual reference clearly uses a large editorial page title.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | #F5F1EA | App background and broad page surfaces |
| Secondary (30%) | #FFFFFF | Cards, input surfaces, content panels |
| Accent (10%) | #5E6B52 | Primary CTA fill, active nav indicator, progress bars, selected chips, focus rings |
| Destructive | #B85C4A | Destructive actions only |

Supporting tokens:

| Token | Value | Usage |
|-------|-------|-------|
| Ink | #1F231E | Primary text and icon strokes |
| Muted text | #6E746A | Secondary copy, metadata, helper text |
| Border | #E7E2D7 | Dividers, card borders, input borders |
| Soft olive | #A8B38E | Secondary badges, subtle selected states |
| Warm clay | #B2765C | Sparing warmth in travel metadata or cost accents, never primary CTA |

Accent reserved for: primary CTA fill, active nav state, selected route/filter chip, progress bars, focus rings, and status badges that indicate active/upcoming trip state.

Do not use accent for every link, every icon, all buttons, or large background fills.

## Layout And Components

Required Phase 1 primitives:

| Primitive | Contract |
|-----------|----------|
| App shell | Responsive header/sidebar foundation with compact navigation, profile/notification affordance placeholders, and centered main content |
| Route inventory | Constants or config mapping only supplied Traveloop screens and flows |
| Page scaffold | Header area, content area, optional toolbar area, and consistent empty/error loading regions |
| Surface card | 8px radius, #FFFFFF background, #E7E2D7 border, soft shadow only when needed for hierarchy |
| Button | 44px minimum height for primary actions, dark olive primary, secondary off-white/white with border, icon+label when command clarity matters |
| Input/search | 44px minimum height, rounded control, lucide icon left when search or email context exists |
| Chip/tag | 32px minimum height, icon optional, reserved for destination categories and filters |
| Navigation item | Active state uses #5E6B52 text or subtle background plus visible indicator |

Radius contract:

- Cards and route panels: 8px.
- Inputs, buttons, chips, and nav controls: 8px unless matching a supplied page image later requires 12px for large photo cards.
- Do not nest cards inside cards.

Responsive contract:

- Mobile shell: single-column content, compact header, bottom nav permitted only when matching supplied mobile references.
- Tablet/desktop shell: horizontal header or left nav depending on the supplied page being implemented later.
- Max readable content width for foundation shell: 1440px.
- Gutter: 16px mobile, 24px tablet, 32px desktop.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | Plan a trip |
| Empty state heading | No trips planned yet |
| Empty state body | Start with a destination and dates, then build your route one city at a time. |
| Error state | We could not load your trip workspace. Check your connection and try again. |
| Destructive confirmation | Delete trip: Type the trip name to confirm permanent deletion. |

Tone: warm, clear, confident, and helpful. Avoid hype, slang, and generic placeholder text. Use sentence case for UI labels except the Traveloop wordmark.

Phase 1 destructive behavior is not implemented, but the confirmation pattern is locked for later trip/profile phases so early primitives support it.

## Implementation Notes For Planner

- Install or plan for `lucide-react` when the frontend package is created.
- Define CSS variables for all color and spacing tokens before page-specific styling.
- Prefer reusable shell primitives over page-specific styling in Phase 1.
- Route placeholders should be database-ready scaffolds, not hardcoded mock workflow screens.
- Preserve supplied image filenames as implementation references in comments or docs when mapping routes, but do not bundle large reference assets into the app unless later phases explicitly need them.
